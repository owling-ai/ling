## 3. Gemini Live 接入与视频模型选型

### 3.1 最初选择

最初使用：

```text
gemini-2.5-flash-native-audio-latest
```

它支持 `bidiGenerateContent`，音频输入输出正常，但视频识别实测不可靠。

测试画面是两行大字：

```text
GEMINI LIVE
STEPFUN
```

2.5 模型虽然接受视频字段，但多次回答成无关内容，例如：

```text
SUMMER / SALE
GOOD MORNING / HAVE FUN
```

结论：协议请求成功不等于模型真的正确使用了视觉上下文，不能仅凭 HTTP/WebSocket 无报错判断视频能力。

### 3.2 视频协议字段

Google GenAI SDK 同时暴露过：

```text
realtimeInput.video
realtimeInput.mediaChunks
```

实测：

- `gemini-3.1-flash-live-preview` 明确拒绝已弃用的 `mediaChunks`。
- 错误信息要求使用 `audio`、`video` 或 `text`。
- 使用 `realtimeInput.video` 后，3.1 模型准确读出测试画面。

最终选择：

```text
gemini-3.1-flash-live-preview
```

最终应用级实测回答：

```text
GEMINI LIVE, STEPFUN!
```

### 3.3 当前视频传输策略

浏览器摄像头帧处理：

```text
最长边：512px
格式：image/jpeg
质量：0.72
频率：1 FPS
```

其他限制：

- WebSocket `bufferedAmount > 512KB` 时跳帧。
- 后端只接受 `image/jpeg`。
- 单帧 base64 最大约 1.2MB。
- 关闭视频、切换 Provider、挂断或离开页面时立即停止摄像头轨道。
- 本地预览镜像显示，但发送给模型的 Canvas 图像保持正常方向。

1 FPS 与 Google Live 官方视频输入建议一致。

## 4. HTTP 代理与 WebSocket 稳定性

今天遇到过两类上游连接问题：

```text
ConnectionResetError
HTTP 200（预期应为 101 Switching Protocols）
```

第二种情况在 Google 官方 `google-genai` SDK 建连时出现，说明当前 HTTP 代理偶尔没有正确透传 WebSocket Upgrade 或 CONNECT。

当前 `websockets==16.0` 支持自动读取：

```text
HTTP_PROXY
HTTPS_PROXY
ALL_PROXY
```

对于 `wss://` 上游，代理必须稳定支持 HTTP CONNECT 和 WebSocket Upgrade。当前应用最终可以连接，但黑客松现场仍应把代理稳定性视为风险项。

## 5. 语音转写为什么与声音不完全一致

### 5.1 Live 附带转写的性质

当前没有调用 Google Cloud Speech-to-Text。我们只在 Gemini Live setup 中启用：

```json
{
  "inputAudioTranscription": {},
  "outputAudioTranscription": {}
}
```

Gemini 在同一个 Live WebSocket 中返回：

```text
serverContent.inputTranscription
serverContent.outputTranscription
```

Google 没有公开内部究竟是同一模型的解码头还是辅助识别器。产品/API 层面应把它视为 Gemini Live 的集成转写能力：

- 不创建 Cloud STT Recognizer。
- 不调用 `speech.googleapis.com`。
- 不单独选择 ASR 模型。
- 官方价格没有单列 transcription SKU。
- 无法保证与原生音频逐字一致。

Gemini 原生音频模型直接生成音频 token。页面文字是对输入或输出音频的附带识别，不是用于合成声音的“原始台词”。

### 5.2 韩文误判

实际出现过：

```text
아니, 테이스터가 강해.
```

这很可能对应用户说的：

```text
No, taste too strong.
```

根因是原 setup 传入空 transcription config，让 ASR 自由猜语种。中英混合短句被错误判成韩语音节。

已增加：

```json
{
  "languageHints": {
    "languageCodes": ["zh-CN", "en-US"]
  },
  "adaptationPhrases": [
    "课程词",
    "中文释义",
    "孩子名字",
    "玩偶名字",
    "本轮分享事件词汇"
  ]
}
```

语言列表可通过以下变量覆盖：

```text
LING_GEMINI_TRANSCRIPTION_LANGUAGES=zh-CN,en-US
```

合成音频实测：

```text
音频：No, taste too strong.
转写：Taste too strong.
模型理解：是吃的东西味道太浓了吗？
```

韩文误判消失，语义理解正确，但开头 `No` 仍可能漏掉。因此集成转写只能改善，不能作为逐字准确字幕。

### 5.3 输入转写提交时机

原实现会在模型刚开始输出时立即提交用户 ASR。后续到达的转写分片可能被拆成第二个气泡，放大“不一致”。

现实现等待本回合 `turnComplete` 后一次性提交完整输入转写，再写入记忆。

### 5.4 是否增加外部 ASR

调研过两个方向：

#### Google Cloud Speech-to-Text V2

官方实时标准价格：

```text
$0.016 / 音频分钟（前 500,000 分钟）
```

优点：

- 当前项目已有 ADC。
- 可复用浏览器上传的 16kHz PCM。
- 与现有 Google Cloud 结算整合简单。

限制：

- 只能直接改善页面字幕和记忆入库。
- Gemini Live 仍然直接理解原始音频，不能自动修正模型已经做出的错误回答。
- 需要流式 partial/final 合并、VAD、重连和长流滚动。

#### 火山大模型流式 ASR

官方协议确认：

- 16kHz PCM S16LE。
- 默认支持中英文和多种中文方言。
- 支持双向流式优化。
- 可开启二遍识别：`enable_nonstream=true`。
- `ssd_version=200` 可启用大模型 SSD 能力。
- 最终结果可带 `definite=true`。

限制：

- 自定义二进制帧、gzip、资源 ID 和额外鉴权。
- 需要新增供应商、账号、计费和数据合规处理。
- 精确价格需要登录火山控制台确认。

最终决策：

```text
当前不接第二路 ASR。
```

先保留 Gemini Live 集成转写、语言提示和动态热词。黑客松阶段不增加不必要的并行流和供应商复杂度。

## 7. 为什么进入聊天页不再自动连接

原行为：

- 进入聊天页立即申请麦克风。
- 立即创建记忆会话。
- 立即连接上游模型。

问题：用户可能只是查看页面，不想马上通话；权限弹窗和上游计费也会无意义发生。

当前行为：

```text
进入页面：只渲染 UI
切换模型：只改变选择
点击接通：申请麦克风 → 创建会话 → 建立 WebSocket
```

Playwright 验证：

```text
打开聊天页：session/start = 0，WebSocket = 0
未接通切换模型：session/start = 0，WebSocket = 0
点击接通：session/start = 1，WebSocket = 1
```

## 8. 开场为什么改成纯问候

原行为会在连接后立即使用昨天记忆钩子，例如：

```text
昨天你说要给三角龙起名字，起好了吗？
```

问题：

- 刚接通就要求孩子回答，交互压力大。
- 容易让孩子感觉系统在执行任务。
- 学习和记忆内容会抢占“我在这里陪你”的基本在场感。
- `engine.start_session` 还会把这条预生成 opening 提前写进数据库，即使模型实际没说，形成“幽灵转写”。

当前规则：

```text
接通第一句：只叫名字、说“我在呢”
禁止：昨天、记忆、学习、待办、故事、问题、要求回答
前两个正常回合：继续跟随孩子
两轮以后：只有自然相关或冷场时，才可轻轻使用一次关系线索
```

实测：

```text
Gemini：悠悠，我在呢！
StepFun：嗨，悠悠，在呢！
```

会话数据库初始 transcript 现在为：

```json
[]
```

真实模型问候到达后才通过 `record_voice_doll` 入库。

StepFun 特殊点：开场需要通过 `response.create.response.instructions` 覆盖本次回复。只写 system prompt 时，StepFun 仍可能主动讲动物园和英语词。

## 9. 冷场后为什么模型不主动说话

### 9.1 根因

Gemini 3.1 默认回合覆盖策略会把视频纳入音频活动形成的回合，但视频帧本身不会触发回答。

Google 官方明确说明：

```text
gemini-3.1-flash-live-preview 不支持 proactiveAudio
```

而且 `proactiveAudio` 主要是让模型在输入不相关时决定“不回答”，不是可靠的视觉变化主动发言器。

因此，原实现除了开场问候外，只能等待用户说话。

### 9.2 已实现的冷场控制

当前加入了应用级 `ling.idle_nudge`：

```text
第一次：连续安静约 20 秒
第二次：至少再间隔 45 秒
每场上限：2 次
```

前端策略：

- 本地麦克风 RMS 检测用户是否正在说话。
- 用户或模型有活动时重置计时。
- 用户正在说话时不触发。
- 模型正在输出时不触发。
- 第一次和第二次之间使用不同冷却时间。

后端策略：

- 在 `engine.SESSIONS` 中保存 `idle_nudges`。
- Provider 切换或 WebSocket 重连不会重置预算。
- 第三次请求由后端拒绝，不仅依赖前端。

第一次冷场指令：

- 只恢复陪伴感。
- 可评论当前画面或问一个容易回答的小问题。
- 禁止昨天记忆、学习议程、待办和复习词。

第二次冷场指令：

- 优先当前画面或刚才话题。
- 只有明显自然相关时才可带出最多一个英语词。
- 禁止宣布学习、考试或强行切话题。

实测：

```text
Gemini：悠悠，你正在画什么呢？
StepFun：在做什么呀？
```

浏览器使用完全静音的虚拟麦克风等待 20 秒，确认页面自动新增模型回复，不依赖手工发送测试事件。

StepFun 特殊点：第二次主动回复必须先创建后台 `conversation.item`，等待 `conversation.item.created` 确认，再发送 `response.create`。直接发送第二个空 `response.create` 不会产生内容。该后台控制消息不写入孩子记忆。

## 10. 视觉“大动作主动回应”调研

用户希望模型看到明显动作变化时主动找话说，但不要过密。

基于本地大动作检测的触发仍未实现。火山 RTC Provider 已实现另一条受控路径：沿用 20 秒 / 45 秒冷场预算，用 `UpdateVoiceChat(ExternalTextToLLM)` 检查服务端缓存画面，`InterruptMode=2` 保证不打断当前交互。

不能直接使用像素差触发回答，因为以下情况都会造成大面积变化：

- 摄像头晃动。
- 光线变化。
- 普通走动。
- 自动曝光变化。
- 背景有人经过。

推荐架构：

```text
低分辨率本地帧差
  → 连续两帧超过阈值
  → 等用户停止说话
  → 轻量视觉模型判断是否值得开口
  → 值得才触发 Live 主动回合
```

建议门控模型：

```text
gemini-3.1-flash-lite
```

官方价格：

```text
输入：$0.25 / 1M text/image/video tokens
输出：$1.50 / 1M tokens
```

按每分钟最多检查两次估算，通常只是每小时几美分以内。

推荐视觉主动预算：

```text
最短间隔：45 秒
单场上限：3 次
用户说话时：排队，不打断
普通走动/光线变化：保持沉默
举起物品/挥手/明显表情或姿态变化：可考虑回应
```

该功能仍是调研结论，不是当前已上线能力。

## 11. 学习主动性的重新定义

两个极端都不合适：

```text
零主动：孩子可能永远不进入学习
强主动：玩偶变成会强行转场的点读机
```

最终方向是“机会型主动”。优先级：

```text
孩子当前需求
  > 当前画面与动作
  > 正在进行的话题
  > 关系记忆
  > 学习议程
```

允许学习引导的三类机会：

| 机会 | 示例 |
|---|---|
| 强相关 | 画面出现猴子玩具，目标词有 `monkey` |
| 弱相关 | 孩子在聊动物园，议程有 `zoo/panda` |
| 冷场 | 话题结束并连续安静，玩偶用自己的生活事件轻轻开新话题 |

建议引导阶梯：

1. 自然曝光：只在普通句子里使用一个词。
2. 轻量邀请：孩子有回应后，问内容而不是考词义。
3. 主动产出：只有孩子明显投入或主动跟读时继续。

建议预算：

```text
首次学习引导：至少两个正常回合后
自然相关机会：每 60 秒最多一次
冷场触发：约 15-20 秒
每场主动学习引导：最多 2 次
每次目标词：最多 1 个
孩子未接住：本场停止主动学习
明确拒绝：立即进入 retreat，整场停止
```

当前已实现的只有：

- prompt 中的跟随孩子规则。
- 第一次冷场禁止学习。
- 第二次冷场仅允许自然相关时最多一个词。
- retreat 规则。

尚未实现：

- 独立 `InitiativeController` 状态机。
- 课程项主题标签。
- 语义相关性评分。
- `ready/offered/engaged/produced/deferred/retreated` 完整议程状态。
- 议程只在真实使用后标记为消费。

当前 `session_agenda` 仍会在会话开始时标记为 `consumed`，这是后续需要修正的设计债务。

## 12. 冷路径 Toast 与总结卡片

现象：挂断后总结卡片最终正常弹出，但“正在整理记忆”的 Toast 有时没有被注意到。

判断：

- 冷路径 API 和总结卡片正常。
- Toast 仅显示约 2.6 秒。
- 页面状态切换、视频区域注意力和异步完成时机都可能让 Toast 被错过。

建议：

```text
关键状态使用通话头部持久状态：正在整理记忆… / 整理完成
Toast 只作为补充
```

该 UI 改造尚未实现。

## 17. 火山引擎 AI 音视频互动方案接入

官方接口版本固定为 `2025-06-01`：

```text
StartVoiceChat   开启 AI 任务
UpdateVoiceChat  主动观察、文本触发和动态更新
StopVoiceChat    让 AI 退出房间
```

媒体不经过 Ling 后端转发。浏览器使用项目内固定的官方 `@volcengine/rtc@4.68.5`：先以短期 Token 加入 RTC 房间并发布麦克风，后端随后调用 `StartVoiceChat` 让 AI 入房。摄像头由用户手动打开，RTC 服务端按 `SnapshotConfig` 抽帧并在用户提问或受控观察触发时送入视觉模型。

后端凭证分为两组：

```text
RTC AppId + AppKey：生成浏览器进房 Token
IAM AK + SK：签名 Start/Update/Stop OpenAPI
```

AppKey、AK、SK 不下发浏览器。浏览器只拿到与当前 `RoomId`、`UserId` 绑定的一小时 Token。当前默认配置：

```text
ASR：火山流式语音识别大模型，双向流式 + 二遍识别
LLM：doubao-seed-2-1-turbo-260628，关闭深度思考，开启 Prefill
TTS：双向流式大模型，zh_female_linjianvhai_moon_bigtts
视觉：360p low detail，1 秒抽帧，单次最近 1 帧
声纹：Mode=1，自动注册 4 秒，EnableSV=true 过滤其他说话人
字幕：SubtitleMode=1，通过 subv 二进制房间消息接收
```

字幕协议为 `subv` Magic Number、4 字节大端负载长度和 JSON。真人 `userId` 映射为 `user`，AI `botId` 映射为 `assistant`，最终分句回传 `engine.record_voice_user/doll`，继续使用同一套记忆、教材编织、撤退和正典逻辑。

生命周期边界：

- `StartVoiceChat` 返回 200 只代表任务下发成功，不保证 AI 已入房；前端继续监听 AI 用户进房/发流事件。
- 挂断时先调用 `StopVoiceChat`，再让真人 `leaveRoom` 并销毁 RTC Engine，避免 AI 或真人留房继续计费。
- 真人异常离线时，`AgentConfig.IdleTimeout` 当前设为 30 秒兜底。
- 火山控制台必须创建「AI 音视频互动方案」应用，不能混用「实时对话式 AI」AppId。

真实云端 AI 入房和字幕已验证。首次实测发现 Seed 2.0 Lite 默认深度思考、每轮两张 480p 视觉输入导致回复明显延迟，且未配置声纹时会收录周围人声；当前已按上述参数切换到 2.1 Turbo 并加入延迟与声纹优化，继续实测。

## 18. 总结

当前产品选型是：

```text
Gemini 3.1 Live：默认音视频交互与视频理解
StepFun Realtime：备用/对照语音 Provider
火山 AI 音视频互动方案：ByteRTC 音视频与方舟视觉 Provider
Gemini Live 集成转写 / 火山 RTC 字幕：各自链路的字幕来源
DeepSeek V4 Pro（SiliconFlow）：冷路径记忆工人
SQLite：五层记忆、议程、正典与会话持久化
```

交互原则已经从“证明我记得你、完成今天的学习议程”调整为：

```text
先建立在场感
再跟随孩子
冷场时有限主动
记忆择机出现
学习只做自然桥接
```

这套方向保留了主动陪伴和学习价值，同时避免把玩偶变成会追问昨天、强行切题或频繁讲话的任务型机器人。
