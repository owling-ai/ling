# 灵 Ling · 记忆架构：事实源、体验投影与数字生命

> 给队友的设计说明。结合最新双端产品设计，讲清楚哪些是长期不变的记忆核心，哪些是黑客松阶段的体验投影与 Mock 实现。技术细节看代码即可（`backend/` 每个文件头都有注释）。

## 一句话总纲

**记忆不是这个产品的一个功能，记忆就是这个产品。**

「共同成长的玩偶」和一个普通语音聊天机器人的全部区别，都落在记忆上：它记得你昨天说过的话（陪伴感）、它记得你三周前怕黑而现在不怕了（成长感）、它自己的世界前后一致（生命感）。所以架构上所有取舍的判断标准只有一个：**这个设计能不能让"它真的记得我"这件事又快又稳地发生。**

最新产品形态增加了孩子端对记忆事实源只读的窗口、公共/专属瞬间、口袋信物和家长端投影，但这些都不应反过来污染记忆事实源。整体分成三层：

```text
实体玩偶会话
  -> 热路径记账
  -> 冷路径整理
  -> L1-L4 / SRS / 私有 Canon                 记忆事实源
       -> 专属瞬间 / 视频 / 信物 / 家庭分享      孩子端体验投影
       -> 今日 / 成长 / 记忆 / 守护 / 通知       家长端体验投影

全局基础世界 -> 时间槽事件 -> 多个等价视频变体 -> 孩子端公共瞬间
```

监护授权、角色权限、红线和账户注销约束所有读写，但不成为新的记忆层。

## 决定一：热路径 / 冷路径分离

**这是整个架构最重要的一刀。**

矛盾在于：实时语音对话要求毫秒级响应，而"好记忆"需要 LLM 慢慢干活（总结日记、提炼事实、反思成长）。把两者塞进同一条链路，要么对话卡顿，要么记忆变浅。

所以切开：

- **热路径（对话进行时）**：禁止一切 LLM 记忆调用。开场时把"记忆包"（孩子卡、玩偶卡、昨日日记、相关事实、今日议程）**一次性从数据库读出来**（毫秒级），注入语音模型的人设 prompt。对话过程中只做轻量记账，不做任何生成式记忆操作。
- **冷路径（对话结束后）**：不赶时间，慢慢用 LLM 干细活——写日记、抽事实、判定学习掌握度、每周反思，并判断是否产生值得发布的专属瞬间候选。视频渲染永远不进入实时语音热路径。

这一刀带来两个额外红利：

1. **记忆系统与语音模型解耦。** 记忆服务只依赖"开场记忆包 + 会后转写"契约；Gemini Live、StepFun 和火山 RTC 共用这一边界，切换实时 provider 不应改变记忆数据模型。
2. **演示保险。** 冷路径的每个 LLM 任务都有纯规则的兜底版本，输出结构一致。在已有转写或 demo 种子数据时，冷路径不依赖外部模型 API 也能跑完；实时语音本身仍需网络和至少一个可用 provider。

## 决定二：记忆分五层，而不是一个大池子

不同的信息，寿命不同、用法不同，混在一起互相污染。分层的本质是**按寿命和用途给信息分家**：

| 层 | 是什么 | 为什么单独一层 |
|---|---|---|
| L0 工作记忆 | 当前对话的最近几轮 | 只活在会话里，用完即弃，不值得落盘 |
| L1 核心卡片 | 孩子卡 + 玩偶状态卡 | **性格稳定性的锚。** 常驻每一条 prompt。没有它，玩偶的人设会随对话漂移——今天胆小明天豪爽，孩子立刻出戏 |
| L2 情景日记 | 每场对话一篇，只增不改 | **一切的原料。** 记忆钩子、家长报告、每周反思全从这里长出来。只增不改（append-only）是刻意的：经历就是经历，不许事后篡改，这是信任的底座 |
| L3 事实记忆 | 关于孩子的稳定事实 | 见决定三，这层藏着产品最好的时刻 |
| L4 反思成长 | 每周一份成长快照 | 日记是流水账，快照是趋势。没有这层，"成长"只是一堆散点，说不出"这个月对恐龙的兴趣升温了" |

对比另一条路——把全部历史对话塞进长上下文：贵、慢、注意力被稀释，而且转写里 99% 是噪音。**策展过的记忆永远好过原始转写。** 分层就是策展的骨架。

这里的 append-only 指普通产品操作中不允许事后改写经历。账户注销触发的数据销毁是明确例外，不能拿 append-only 当作继续保留儿童数据的理由。

## 决定三：事实演化、红线和数据销毁必须分开

L3 的事实带两个字段：生效时间、被谁替代（`superseded_by`）。孩子说"我不怕黑了"，我们不是把"怕黑"删掉，而是插入新事实、把旧事实标记为"已被替代"。

为什么这么绕？因为**成长感恰恰藏在被作废的旧事实里**。

家长控制台上那一行「以前：有点怕黑，睡觉要开灯 → 现在：不怕了，因为有了恐龙小夜灯」，是整个产品最打动人的画面。如果当初直接覆盖，这个时刻就永远消失了。玩偶也能用它说出"你以前还怕黑呢，现在都自己关灯睡觉啦"——这句话只有"记得旧版本的你"的朋友才说得出来。

但三个看起来都像"删除"的动作，语义完全不同：

- **事实演化**：新增事实，旧事实进入历史版本。它仍可用于成长对比，但不再作为当前事实召回。
- **红线**：保留原始经历，只禁止未来对话、瞬间、提醒和记忆钩子再次主动提起。红线是召回策略，不是记忆手术。
- **账户注销（长期不变量）**：这是极端且主动的彻底销毁行为。普通界面不提供策展式逐条删除。生产版在法务确认范围后，先把账户标记为 `deletion_pending` 并停止所有召回和生成，再硬删除私有转写、日记、事实、掌握度观察、私有 Canon、专属瞬间、私有媒体关联、信物、口袋状态、生成任务和家庭分享。最后只保留不含儿童内容的操作 tombstone，防止任务重试把内容重新生成出来。黑客松只展示这个入口和语义，不声称已完成全量级联销毁。

把原始内容长期搬进一张"已删除表"不算销毁，也会扩大泄露面。普通界面不提供家长挑着删来改写叙事的能力；数据权利的生产范围仍需上线前由法务确认。

## 决定四：关系线索是"预生成"的，但不抢占开场

孩子接通时，玩偶第一句话只做纯问候：叫孩子名字、说「我在呢」，不提昨天、不问问题，也不要求孩子立刻接住一段记忆。至少经过两个正常回合后，关系线索才可以在自然相关或冷场时轻轻出现。

直觉做法是现场做一次检索甚至 RAG。我们没有。**夜间规划器每天睡前就把第二天可用的关系线索生成好了**，写进今日议程表。热路径仍然只是一次普通数据库读，但「读入」不等于「马上说出」。

为什么：

- 刚接通先建立在场感，比证明「我记得你」更自然。
- 回忆的质量在于**时机和选择都准**，不在于搜得广。昨天日记里那个未完成事件可以成为线索，但不能成为开场任务。
- 孩子带来自己的话题时，关系线索继续等待；整场没用也不算失败。
- 玩偶仍然显得「睡前还在想你」，但不会一接通就向孩子追问。

同理，我们没用向量数据库。demo 尺度下事实总量很小，结构化存储 + 直接注入已经够准，引入向量库只增加依赖和故障面，不增加"记得你"的体感。等事实量到了千条级，再换不迟——分层架构保证到时候只用换 L3 的检索方式，其他不动。

## 决定五：复习藏在"玩偶自己的生活"里

英语复习是商业化的钩子，但也是最容易毁掉产品的东西——一旦孩子觉得"这是个会考我的点读机"，陪伴关系就死了。

所以复习内容从不直接出现，而是**寄生在下一次会话的生活叙述里**：夜间规划器从掌握度表挑出到期该复习的词，只把它们写进会话议程。下次对话中，如果孩子的话题自然相关，玩偶才借当前基础事件或已有私有故事上下文带出它（"我今天去动物园 zoo 送请柬，见到了 panda，就是熊猫呀"）。这份议程不修改全局基础事件，也不会在孩子离线时生成专属瞬间。

配套三条纪律，全部是产品判断而不是技术限制：

- **密度上限**：一场对话最多自然带出 3 个词。超了就是上课。
- **机会主义触发**：只有孩子主动聊到相关话题才顺势带词，绝不硬转话题。
- **撤退规则**：孩子说"不想说英语"、表现出敷衍——立刻放弃今天全部复习议程，纯陪伴。**宁可今天一个词都不复习，也不消耗关系。** 这是玩偶和点读机的分界线，也是我们敢把"学习"放进陪伴产品的底气。

学习效果的判定也不打扰孩子：会话结束后冷路径从转写里判定每个词到了哪一层（听到过 → 听懂了 → 会说了），按记忆曲线安排下次出现的时间。孩子全程无感，家长在报表里看到曲线。

## 决定六：数字生命分成全局基础世界和孩子私有覆盖层

玩偶的数字生命不再是单一事件流，而是两条来源在孩子端合并展示。

### 全局基础世界

所有灵灵共享同一套基础事件库、语义时间槽和作息。系统按孩子所在时区解析当前时间槽；家长设置的可用/睡前时段只控制孩子能否访问和互动，不改写基础世界作息。例如同一时间槽的语义事件都是"早上去山坡等风"，但可以预生成多份不同镜头、动作和场景细节的视频，稳定分配给不同批次的玩偶。

硬约束：

- `base_world_event` 是唯一语义事实源；视频只是它的渲染变体。
- 同一事件的所有视频变体必须表达同一件事，不能擅自新增故事事实。
- 变体按 `{doll_id, event_id, event_version}` 稳定分配。仅在发布前可降级到同一事件的其他变体；发布后固定 `variant_id`，素材异常时隐藏并报错，不悄悄换片。
- 基础事件不读取孩子的私有 Canon，内容制作必须避开可能被孩子改写的细节。
- `event_id` / `event_version` / `variant_id` 一旦发布就只能追加新版本，不就地覆盖，保证已发布的公共瞬间也可复现。
- 全局基础世界不复制进每个孩子的记忆库。不含儿童信息且未绑定个体账户的公共素材不随单个账户销毁；个体分配、观看记录和其他私有派生记录仍进入数据权利流程。

黑客松阶段使用版本控制内的 `base_world.json` 描述时间槽和语义事件，用素材 manifest 描述视频变体。生产阶段再考虑 CMS、对象存储和分发策略。

### 孩子私有覆盖层

孩子的共同经历继续使用现有 L1-L4，并增加三样数字生命能力：

- **世界正典（Canon）**：玩偶私有世界的"设定账本"。实体玩偶对话和专属投影必须与它一致；全局基础事件则不读取它。为什么必须有：生成式模型自由发挥，三天就会自相矛盾——秋千一会儿蓝色一会儿红色，孩子比谁都先发现穿帮。正典就是防穿帮的账本。
- **故事弧**：玩偶的生活不是随机事件，而是有骨架的连续故事（比如"帮松鼠先生筹备生日会"，五拍推进）。随机事件没有期待感，连续故事才让孩子想着"后来怎么样了"。
- **私有成长时钟**：记录共同经历的日期和故事状态，只有孩子的有意义互动或确认选择才推进私有故事拍。孩子不在时的"灵灵也在生活"由全局基础世界承担，不凭空生成孩子的专属经历。

最关键的一环：故事推进到需要做决定的地方，玩偶把难题抛给孩子（"生日蛋糕做橡果味还是蜂蜜味？"），**孩子的选择写进私有 Canon**。从此这个孩子的灵灵在普通产品操作中始终记得蛋糕是橡果味的；账户注销仍可依数据权利流程级联销毁。私有 Canon 只影响实体玩偶对话、专属瞬间、口袋和家庭视图，不反向修改全局基础视频。若展示层发现两者意外冲突，对该孩子隐藏基础事件并报错，不改写任何一边的事实。

## 决定七：瞬间是体验投影，不是 L5 记忆

专属瞬间来自记忆事实源，但它本身是面向孩子发布的产品对象。把它直接塞进日记或生活事件会混淆"发生了什么"与"如何讲给孩子看"。

生成规则：

- 每天 0-3 个，3 是硬上限，不是必须填满的指标。
- 只有孩子做出世界决定、主动教会灵灵一个词、完成故事节点或出现明确成长变化时，才产生 `moment_candidate`。
- 普通问候、短对话、重复内容和未通过安全检查的候选不生成瞬间，也不占配额。
- 同一来源会话和事件类型必须有幂等键，重复结算不能产生两条瞬间。
- demo 的 `moment` 保存语义、来源、故事文本和 `published_asset_id`；生成尝试另存在 `generation_jobs`，不新增 `moment_render` 表。
- 视频在发布前可以重试，发布时原子写入 `published_asset_id`。发布后语义、故事文本和资产 ID 在普通操作中不可修改；模型升级不允许静默重绘孩子已经看过的回忆，账户注销仍可级联销毁。

口袋信物同样不是记忆层，也不是奖励系统。只有少量专属瞬间自然产生一个信物；孩子点击"收进口袋"只是收藏。`keepsake` 保存名称、外观和来源瞬间，`pocket_entry` 只记录收藏状态。取消收藏只将状态改为 `collected=false`，不删行、不删信物、不改写瞬间；上述对象仍受账户注销级联约束。

## 决定八：一份事实源，多个受控投影

实体玩偶、孩子 App 和家长 App 共用同一记忆事实源，但不再直接读取同一组裸表：

- **实体玩偶**是主要写入入口：语音转写、孩子决定、掌握度观察和私有 Canon 都从这里产生。
- **孩子 App**对记忆事实源只读：只看基础世界、专属瞬间和口袋，不聊天、不看掌握度、不看家长洞察；唯一可写的是收藏与否这类体验投影状态。
- **家长 App**读取受控投影：今日、成长、统一记忆时间线、守护设置和数据权利入口。

为什么：记忆是资产，客户端是投影。"同一份"指同一事实源和可追溯来源，不代表把原始转写、学习议程或所有内部字段无差别暴露给每个端。受控客户端投影不限制独立数据权利流程依法导出或删除更完整的数据范围。

## 决定九：黑客松现场永远不依赖真实视频生成

视频和图片采用同一生命周期契约。业务层只依赖 `GenerationProvider`，`MockVideoProvider`、`SeedanceVideoProvider`、`VeoVideoProvider` 和图片 provider 都是适配器：

```text
GenerationProvider.submit(request) -> job_id
GenerationProvider.poll(job_id) -> queued | running | succeeded | failed
GenerationProvider.result(job_id) -> asset
```

`request` 至少包含 `media_kind`、`event_key`、来源 ID、语义版本、幂等键和允许的素材组；`asset` 统一返回 `asset_id`、URI、MIME、尺寸、时长/poster 和 provenance。`result()` 在未完成、失败和任务不存在时分别返回 `not_ready`、`generation_failed` 和 `not_found`，业务层不解析任何厂商私有响应。

黑客松完成定义只包含 `MockVideoProvider`：它根据 `word_taught`、`canon_choice`、`story_beat`、`growth_change` 等事件键选择预生成素材，模拟 2-4 秒生成状态，再返回固定视频。现场演示只覆盖 3-5 个经过排练的高质量场景；未命中有意义事件时宁可不生成，也不返回不相关视频。

Seedance、Google Veo 或其他真实模型通过同一接口接入，只是加分项。可以离线生成素材并通过导入命令登记进 manifest，但真实 API 的耗时、网络、费用和内容漂移都不能决定现场演示是否成功。

黑客松阶段建议只新增以下持久化对象：

| 对象 | 作用 |
|---|---|
| `moments` | 专属瞬间语义、来源、事件类型、发布状态和 `published_asset_id` |
| `generation_jobs` | 媒体类型、provider、状态、`ready_at`、素材 ID、错误、幂等键和尝试次数 |
| `keepsakes` | 少量瞬间自然产生的故事信物 |
| `pocket_entries` | 孩子对信物的收藏状态 |
| `base_world.json` | 全局时间槽与统一语义事件 |
| `mock_assets.json` | 预生成视频/图片及事件键映射 |

为了控制工期，demo 只新增这组表和配置文件，不改造旧表，不建设真实任务队列、CMS、多 provider 调度或完整账户体系。生产阶段再补数据库迁移、外键、监护人/家庭 ACL、持久任务、对象存储、通知 outbox 和完整注销级联。

### 黑客松最小运行契约

```text
候选不通过 -> 不创建 moment
候选通过   -> moment.rendering -> moment.published | moment.failed

job.queued -> job.running -> job.succeeded | job.failed
```

- 安全检查、事件意义判定和素材命中必须在占用配额前完成。按孩子配置时区计算日界线；demo 使用单一配置时区。
- `moments.idempotency_key` 唯一，至少由孩子、来源会话/事件、事件类型和语义版本组成。配额检查、`moment.rendering` 插入和首个 job 必须在一个事务内完成；`rendering + published` 最多 3 条，终态 `failed` 释放名额。
- 重复结算返回已有 moment/job。重试只给同一 moment 增加 job attempt，最多两次，不复制 moment。`keepsakes.moment_id` 和 `pocket_entries(child_id, keepsake_id)` 也必须唯一。
- Mock job 用数据库中的 `created_at` / `ready_at` 推导 2-4 秒状态，不依赖进程内定时器，服务重启后仍可继续。
- manifest 未命中时不创建 moment；文件缺失、校验失败或超时时标记 job/moment 失败并从孩子 feed 隐藏。绝不用不相关视频兜底。

demo 对客户端只暴露业务 API，provider 不直接暴露：

| API | 用途 |
|---|---|
| `POST /api/session/end` | 幂等结算会话，返回新建或已有的 moment/job 状态 |
| `GET /api/moments/{id}` | 读取 2-4 秒生成状态和发布结果 |
| `GET /api/child/world/now` | 按时间槽返回当前基础事件和稳定视频变体 |
| `GET /api/child/feed` | 合并公共瞬间与已发布专属瞬间 |
| `GET /api/pocket` | 返回已收藏信物 |
| `PUT /api/pocket/{keepsake_id}` | 幂等设置 `collected=true|false`，不删除任何记忆对象 |

`base_world.json` 至少包含 schema/world 版本、稳定事件 ID、时间槽、语义摘要和素材组；`mock_assets.json` 至少包含唯一资产 ID、事件键/素材组、视频与 poster 路径、时长、MIME、尺寸、校验值和语义版本。启动和测试时必须验证所有引用存在。

demo 验收只看一条可离线复现的闭环：种子数据 -> 触发 `word_taught` 或 `canon_choice` -> 重复结算仍只有一个 moment -> 2-4 秒后 feed 可见 -> 收藏后 pocket 可见。同时验证每日不超过 3 条、服务重启可续跑、素材缺失不发布、已发布资产不被 manifest 更换、移动端视频/poster 实际可播放且不溢出。

## 快速问答（预判你们会问的）

**Q：为什么不直接用超长上下文，把历史全塞进去？**
贵、慢、人设会被海量噪音稀释；而且"记得"的产品体感来自策展（挑对那一句），不来自覆盖（什么都在）。

**Q：为什么不用向量库 / RAG？**
当前尺度用不上，徒增依赖。分层设计保证以后要换只动 L3 检索，见决定四。

**Q：换语音模型 / 上硬件，记忆要重做吗？**
不用。记忆系统只依赖"开场注入记忆包 + 结束拿到转写"两个接口，对中间的对话形态无感。见决定一。

**Q：没有冷路径模型 API key 能演示吗？**
可以。已有转写或种子数据时，日记、事实、反思和规划都可用规则兜底，视频/图片使用 Mock 素材。实时语音仍需 Gemini、StepFun 或火山 RTC 中至少一个可用凭证与网络。
